# Концепция и формат учебных интенсивов

## Назначение

Репозиторий содержит готовые учебные маршруты по структуре базы знаний `knowledge`.

Интенсивы предназначены для практического освоения областей знаний, а не для хранения итоговой теории:

- база знаний хранит законченные и обобщённые знания;
- репозиторий интенсивов хранит личные маршруты, задания, лабораторные работы и практические результаты;
- база знаний асинхронно пополняется по мере прохождения.

Интенсив должен позволять начать работу без самостоятельного проектирования программы, поиска основных источников и придумывания заданий.

## Целевой результат

Основной интенсив по разделу:

- покрывает все основные подразделы;
- позволяет ответить на основные вопросы раздела;
- обеспечивает уровень 3 по всей области;
- обеспечивает уровень 4 по ключевым механизмам;
- содержит практику, проверки и итоговую интеграционную работу;
- приводит к созданию или улучшению заметок в `knowledge`.

Уровень 3 означает самостоятельное применение в знакомых ситуациях. Уровень 4 означает анализ, диагностику, сравнение альтернатив и адаптацию знаний к изменённым условиям.

## Разделение ответственности

При создании интенсива заранее подготавливаются:

- последовательность спринтов;
- цели и результаты;
- основной, альтернативный и углублённый набор ресурсов;
- конкретные задания;
- лабораторные работы;
- диагностические сценарии;
- проверочные спринты;
- итоговая работа;
- критерии завершения;
- рекомендации по развитию базы знаний.

Во время прохождения пользователь изучает материалы, выполняет практику, сохраняет полезные артефакты и переносит обобщённые знания в `knowledge`.

## Минимальная бюрократия

Репозиторий не является LMS или task tracker. Порядок задаётся каталогами и нумерацией.

Не ведутся обязательные backlog, статусы, проценты, сроки, журналы действий, отчёты о времени и таблицы синхронизации.

Сохраняются только содержательные результаты: код, тесты, конфигурация, измерения, трассировки, схемы, диагностика и заметки базы знаний.

## Иерархия

```text
раздел базы знаний
├── основной интенсив
│   ├── содержательные спринты
│   ├── проверочные спринты
│   └── итоговый спринт
└── дополнительные тематические интенсивы
    ├── содержательные спринты
    ├── проверочные спринты
    └── итоговый спринт
```

## Раздел

Раздел соответствует верхнему уровню базы знаний. Его `README.md` является статическим каталогом будущих интенсивов и описывает назначение области и связи с соседними разделами.

## Основной интенсив

Основной интенсив формирует целостную модель предметной области. Он охватывает все обязательные темы и не превращается в бесконечный экспертный курс.

## Дополнительные интенсивы

Углубления создаются по конкретным направлениям: например, «Память и производительность», «Linux internals», «Маршрутизация», «Транзакции и согласованность».

Каждое углубление имеет собственный результат, предварительные требования и итоговую работу.

## Спринт

Спринт является минимальной законченной единицей прохождения. Обычно он занимает 5–10 часов и отвечает на один центральный вопрос.

Спринт включает:

- проверяемые результаты обучения;
- место в общей последовательности;
- точный маршрут изучения;
- организованный набор ресурсов;
- небольшие практические задания;
- основную лабораторию;
- изменённый эксперимент или диагностику;
- самостоятельную проверку;
- рекомендации для базы знаний;
- критерии перехода дальше.

Спринт не является формой для заполнения. Его `README.md` — полностью готовая инструкция.

## Ресурсы

Ресурсы группируются по назначению:

- обязательный маршрут — минимально достаточный набор;
- альтернативные объяснения — замена непонятному источнику;
- углубление — материал для уровня 4;
- справочные материалы — документация для точечной проверки.

Для книг указываются главы, для видео — нужные фрагменты, для документации — конкретные разделы.

## Практика

Практика должна исследовать механизмы, а не интерфейс инструмента. Пользователь не придумывает продолжение задания самостоятельно.

Обычный спринт содержит несколько небольших задач, одну основную лабораторию, дополнительный эксперимент и самопроверку.

## Проверки

После каждых 3–5 спринтов добавляется проверочный спринт без новой теории. Он проверяет объяснение, применение и диагностику.

## Итоговая работа

Итог объединяет несколько тем, даёт воспроизводимый результат, содержит измерения, подготовленную неисправность и её диагностику.

## База знаний

В конце спринта указываются рекомендуемые заметки, их вопросы, границы и связи. Отдельный backlog не создаётся.

Обобщённое знание находится в `knowledge`. Конкретный код и результаты эксперимента находятся рядом со спринтом.

## Технологии

Основной практический стек: Go, Linux, Bash, Docker или Compose и Git. Другие языки и инструменты применяются, когда они лучше раскрывают механизм.

## Критерий успеха

Пользователь выполняет только содержательную работу:

1. изучает подготовленные материалы;
2. выполняет готовые задания;
3. сохраняет практические результаты;
4. переносит полезные обобщения в базу знаний.

Планирование, поиск основных источников, формулировка заданий и критериев проверки реализуются заранее внутри репозитория.
