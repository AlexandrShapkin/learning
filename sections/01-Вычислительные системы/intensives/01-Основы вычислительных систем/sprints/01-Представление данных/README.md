# Спринт 01. Представление данных

Ориентировочный объём: 8–12 часов.

## Центральный вопрос

Как один и тот же набор битов превращается в число, текст, инструкцию или часть структуры данных?

## Результат спринта

После завершения необходимо уметь:

- переводить значения между двоичной, шестнадцатеричной и десятичной системами;
- объяснять диапазоны целых типов и дополнительный код;
- предсказывать целочисленное переполнение в модели фиксированной разрядности;
- объяснять структуру IEEE 754, ошибки округления, `NaN`, бесконечности и отрицательный ноль;
- определять UTF-8-представление строки;
- различать значение, его тип и конкретное расположение байтов;
- объяснять little-endian и big-endian;
- проектировать простой устойчивый бинарный формат.

## Место в интенсиве

Все последующие спринты опираются на представление данных. Машинная инструкция является битовой строкой, адрес — целым числом, floating point влияет на производительность и корректность, а порядок байтов проявляется в файлах, сетевых протоколах и памяти.

## Предварительные знания

Прочитать CS:APP, глава 2:

До перехода дальше самостоятельно вывести диапазоны `uint8`, `int8`, `uint16`, `int16`, а не запоминать их.

### Шаг 2. Floating point

Прочитать CS:APP 2.4 целиком. Сосредоточиться на полях sign/exponent/fraction, нормализованных и денормализованных значениях, округлении и специальных значениях.

Не требуется вручную выполнять сложную арифметику IEEE 754. Требуется уметь разобрать конкретное значение по полям и объяснить потерю точности.

### Шаг 3. Текст и кодировки

Прочитать:

- Go Blog, «Strings, bytes, runes and characters»: https://go.dev/blog/strings
- Unicode UTF-8 FAQ: https://www.unicode.org/faq/utf_bom.html

Разобраться, почему длина строки в байтах, количество Unicode code points и количество пользовательских символов могут различаться.

### Шаг 4. Порядок байтов и сериализация

Изучить пакеты:

- https://pkg.go.dev/encoding/binary
- https://pkg.go.dev/math#Float64bits

Сформировать модель: тип задаёт интерпретацию значения, byte order задаёт порядок хранения многобайтового значения, формат задаёт договор между производителем и потребителем данных.

## Практические задания

### 1. Таблица представлений

Без программ сначала заполнить таблицу для значений:

`0`, `1`, `15`, `127`, `128`, `255`, `-1`, `-128`, `-129`.

Для каждого значения указать:

- hex-представление;
- 8-битное unsigned-представление, если возможно;
- 8-битное signed two's complement, если возможно;
- 16-битное signed-представление.

После этого написать тест, который проверяет допустимые строки таблицы. Недопустимые значения должны быть явно отмечены, а не молча усечены.

### 2. Прогноз переполнения

До запуска программы предсказать результаты в 8-битной модели:

- `127 + 1`;
- `255 + 1`;
- `-128 - 1`;
- `200 * 2` как unsigned;
- сравнение одного байта `0xff` как `uint8` и `int8`.

Затем проверить прогноз отдельной программой или тестами. В Go явное преобразование к `uint8`/`int8` должно быть видно в коде.

### 3. Разбор floating point

Для значений `0`, `-0`, `0.1`, `0.2`, `0.1+0.2`, `1.5`, `+Inf`, `NaN` вывести:

- 64 бита в binary и hex;
- sign, exponent и fraction;
- классификацию значения;
- соседние значения через `math.Nextafter` для `0.1` и `1.0`.

Объяснить, почему сравнение `NaN == NaN` ложно и почему `0.1 + 0.2` нельзя рассматривать как точную десятичную арифметику.

### 4. UTF-8

Для строк `hello`, `Привет`, `€`, `👨‍💻`, `é` вывести:

- количество байтов;
- количество runes;
- hex каждого байта;
- Unicode code point каждой rune.

Последняя строка должна быть проверена в двух вариантах: готовый `é` и комбинация `e` + combining acute accent.

## Лабораторная работа

Создать в `work/repr/` CLI-программу `repr` на Go.

Обязательные команды:

```text
repr int --bits 8 --signed 127
repr float 0.1
repr text "Привет"
repr encode --order little record.json record.bin
repr decode --order little record.bin
```

### Требования

1. `int` показывает decimal, binary, hex, диапазон типа и результат интерпретации тех же битов со знаком и без знака.
2. `float` разбирает `float64` на sign/exponent/fraction и показывает соседние значения.
3. `text` показывает bytes, runes и UTF-8 validity.
4. `encode/decode` работают с собственным бинарным форматом записи:
   - magic: 2 bytes;
   - version: 1 byte;
   - flags: 1 byte;
   - timestamp: uint64;
   - temperature: float32;
   - payload length: uint16;
   - UTF-8 payload.
5. Парсер проверяет magic, version, длину и корректность UTF-8.
6. Есть unit tests на round trip, повреждённую длину, неправильный magic и обе byte order.

### Проверка

```bash
go test ./...
go run . encode --order little record.json /tmp/record.bin
xxd /tmp/record.bin
go run . decode --order little /tmp/record.bin
```

Round trip должен восстанавливать исходные поля. Декодирование little-endian-файла в big-endian-режиме должно либо завершаться понятной ошибкой, либо давать явно объяснимое неверное значение; молчаливое принятие запрещено.

## Дополнительный эксперимент

Написать C-программу, которая выводит байты `uint32_t value = 0x01020304` через указатель `unsigned char *`. Сравнить фактический порядок с результатом `binary.NativeEndian` в Go.

Затем вручную переставить байты бинарного файла лаборатории и установить, какая проверка обнаруживает повреждение первой.

## Самопроверка

Без материалов ответить:

1. Почему `0xff` может означать `255`, `-1`, часть инструкции или байт UTF-8?
2. Как получить диапазон signed N-bit integer?
3. Почему арифметический и логический shift вправо различаются?
4. Почему равные десятичные строки могут иметь разные Unicode-последовательности?
5. Почему byte order не имеет смысла для отдельного байта?
6. Какие поля должен содержать бинарный формат, чтобы безопасно развиваться?

Практическая проверка: получить неизвестный 16-байтовый hex dump собственного формата и вручную определить каждое поле до запуска декодера.

## Результат для базы знаний

Рекомендуется создать или дополнить:

- **Биты, байты и машинное слово** — связь единиц хранения и разрядности; связать с архитектурой процессора и адресацией.
- **Целые числа** — unsigned, two's complement, диапазоны и переполнение; не включать особенности конкретного языка сверх примеров.
- **Числа с плавающей точкой** — IEEE 754, точность, округление и специальные значения; связать с производительностью и сериализацией.
- **Кодировки и UTF-8** — bytes, code points, grapheme clusters и нормализация; связать с представлением данных.
- **Порядок байтов** — little/big endian и границы применения; связать с файлами, сетью и машинными инструкциями.
- **Бинарная сериализация** — versioning, length, validation и byte order; не превращать заметку в документацию лабораторного формата.

## Когда переходить дальше

Можно переходить, когда программа и тесты проходят, любой байтовый dump лабораторного формата объясним вручную, а основные вопросы самопроверки отвечаются без обращения к определениям.
