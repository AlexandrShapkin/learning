# Спринт 06. Packages, modules и зависимости

Ориентировочный объём: 12–18 часов.

## Центральный вопрос

Как разделить codebase на compilation/API units, получить third-party code воспроизводимо и управлять version compatibility?

## Результат спринта

После завершения необходимо уметь:

- distinguish package, module, repository and distribution artifact;
- organize Go code by responsibility rather than arbitrary technical layers;
- understand import paths and `internal` visibility;
- design small stable exported API;
- use `go.mod`, `go.sum`, `go get`, `go mod tidy`, `go mod graph`, `go mod why`;
- explain minimal version selection at practical level;
- use semantic versioning and Go major-version import paths;
- distinguish direct/indirect dependencies;
- inspect module cache and verify checksums;
- use `replace`/workspace only for deliberate local development;
- avoid hidden dependency through global state/init;
- understand vendoring trade-offs;
- evaluate dependency by maintenance, license, API, transitive graph and security posture;
- produce repeatable offline-ish build after dependencies are fetched;
- avoid unnecessary third-party packages when standard library suffices.

## Основной маршрут

### Шаг 1. Package organization

Read Go docs on package naming, `internal`, commands and modules. Use package dependency direction visible through imports.

### Шаг 2. Module graph

Study Go modules reference:

- module path/version;
- `require`, `replace`, `exclude`, `retract`;
- checksums and proxy/direct resolution;
- minimal version selection;
- `go mod tidy` behavior.

### Шаг 3. Version compatibility

Use SemVer carefully: version numbers express intent, not proof of compatibility. Test upgrade with public API and behavior tests.

### Шаг 4. Dependency review

For each proposed dependency answer:

```text
What capability is needed?
Can stdlib provide it?
What API surface is imported?
How large is transitive graph?
How is version pinned/verified?
How will upgrade/removal be tested?
```

## Практические задания

### 1. Module graph inspection

Take one small dependency and inspect:

```bash
go list -m all
go mod graph
go mod why -m MODULE
go mod download -json MODULE
```

Identify direct/indirect modules and why each exists.

### 2. Local version evolution

Create library versions `v0.1.0`, `v0.2.0`, incompatible `v1.0.0` in local Git repos/tags. Upgrade consumer and use tests to identify source/behavior compatibility.

### 3. Replace misuse

Demonstrate local `replace` making build pass while clean checkout fails. Add CI/check script that rejects unintended replace directives.

### 4. Package cycle

Create then remove an import cycle by moving shared abstraction to consumer or reorganizing responsibility. Do not create generic `common` dumping ground.

## Лабораторная работа

Создать `work/recordkit/`:

```text
recordkit/
├── cmd/recordctl/
├── internal/app/
├── internal/storage/
├── pkg-or-exported-package-only-if-justified/
├── go.mod
└── tests/
```

### Requirements

- reusable parsing/validation library with narrow API;
- CLI consumer;
- second small external consumer in separate module;
- one third-party dependency with documented reason;
- stdlib alternative evaluated;
- module graph report generated by script;
- build with `GONOSUMDB`/private settings not required;
- test upgrade and rollback of dependency;
- no import cycles/global init side effects;
- deterministic version output via build info where possible;
- clean checkout instructions;
- optional `vendor` experiment, vendor directory removed afterward unless explicitly retained.

### Failure scenarios

1. Missing `go.sum` entry.
2. Network unavailable after empty module cache.
3. Incompatible dependency API upgrade.
4. Transitive behavior change.
5. Accidental `replace` to local path.
6. Module path/major version mismatch.
7. Init order hidden dependency.
8. Package export leaks implementation type.

## Дополнительный эксперимент

Compare Go module graph/version selection with npm lockfile or Cargo.lock on tiny project. Focus on resolution semantics and reproducibility, not ecosystem ranking.
