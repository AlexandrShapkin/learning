# Спринт 10. Runtime, ABI, reflection и program startup

Ориентировочный объём: 13–19 часов.

## Центральный вопрос

Что происходит между entry point executable и пользовательской функцией `main`, и какие hidden services предоставляет runtime во время выполнения?

## Результат спринта

После завершения необходимо уметь:

- explain program startup from loader entry to runtime initialization and `main`;
- distinguish language runtime, standard library and OS kernel;
- explain calling convention/ABI at practical level;
- read stack frames, arguments/results and return addresses in debugger/disassembly;
- understand Go runtime roles: goroutine stacks, scheduler, GC, panic, maps, channels, reflection, timers and network poller overview;
- inspect process mappings, threads and goroutines;
- understand package initialization ordering and `init` risks;
- explain interface/type metadata and reflection conceptually;
- use `reflect` safely and identify compile-time guarantees lost;
- inspect build/runtime version and module metadata;
- distinguish native thread, goroutine and runtime scheduler;
- understand FFI/cgo boundary and ABI responsibility;
- diagnose startup panic, init-order issue, stack trace and reflection misuse.

## Основной маршрут

### Шаг 1. Startup sequence

Trace:

```text
execve → ELF loader/dynamic loader → runtime entry
→ runtime initialization → package initialization → main.main
```

Use `strace`, debugger breakpoints and symbols as available. Exact internal function names vary by Go version.

### Шаг 2. ABI and call frames

Study target System V ABI overview and Go internal ABI at high level. Inspect one function call in disassembly and debugger:

- arguments/registers/stack;
- return address;
- stack frame;
- saved state;
- stack trace metadata.

### Шаг 3. Runtime services

Read Go runtime package and diagnostics docs. Map source constructs to runtime support without assuming every operation calls runtime function.

### Шаг 4. Reflection and metadata

Study `reflect`, interfaces and struct tags. Prefer static interfaces/generics when shape is known; reflection is justified for serializers, dependency frameworks and generic inspection where runtime types are inherent.

## Практические задания

### 1. Startup trace

Use `strace -f`, GDB/Delve and `/proc/PID/maps` to capture selected startup events. Identify executable mapping, libraries, threads and transition to `main`.

### 2. Package init

Create packages with explicit observable initialization, then remove hidden ordering dependency. Use tests to show why `init` complicates configuration/error handling.

### 3. Call frames

Debug a chain of non-inlined functions. Inspect arguments, locals, stack trace and return. Repeat with optimizations and explain missing/moved variables.

### 4. Reflection

Implement generic struct inspector/decoder with reflection and compare to explicit typed decoder. Test unexported fields, tags, nil pointers, type mismatch and panic-prone operations.

## Лабораторная работа

Create `work/runtime-probe/`.

### CLI

```text
runtime-probe buildinfo
runtime-probe process
runtime-probe goroutines
runtime-probe stack
runtime-probe types VALUE
runtime-probe init-trace
runtime-probe cgo-call   # optional
```

### Required output

- Go/runtime/OS/arch/build/module versions;
- PID, mappings summary, OS thread count;
- goroutine profile summary without dumping unrelated secrets;
- current/all-goroutine stacks in controlled test;
- selected type metadata and method sets;
- init sequence generated by fixture packages;
- optional C call boundary timings/stack trace.

### Requirements

- JSON and text output;
- deterministic fixtures;
- no unsafe runtime internals as mandatory path;
- reflection errors returned, not panics;
- tests for typed nil, pointer/value, tags and unsupported kinds;
- build with inlining disabled for frame lab only;
- cleanup child processes.

### Failure scenarios

1. Panic during package initialization.
2. Init depends on environment unavailable at startup.
3. Reflection calls `Elem`/`Interface` on invalid/unexported value.
4. Plugin/FFI symbol or ABI mismatch optional.
5. Optimized frame hides variable.
6. Stack trace lacks source due stripped/debug metadata.
7. Runtime metadata/version differs from expected build.
8. Goroutine stack dump is mistaken for OS thread stack.

## Дополнительный эксперимент

Use `GODEBUG=schedtrace=...` or runtime trace on small workload and map goroutines to OS threads. Avoid relying on unstable textual format as API.
