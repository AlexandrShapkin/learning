# Итоговая проверка

## Blind engineering exercise

Solve at least 24 hidden cases:

- four framing/requirements cases;
- four priority/forecast/flow cases;
- four Git/review/ownership cases;
- four quality/CI/defect cases;
- four documentation/decision/knowledge cases;
- four metrics/improvement/cross-layer cases.

For each case:

```text
Exact outcome, quality or coordination symptom
Change, requirement, branch, review and release scope
Preserved requirements, commits, tests, decisions and metric definitions
Timeline of active work, waiting, rework and decisions
Facts, hypotheses and one discriminating read-only check
First divergence from intended process and contributing conditions
Minimal correction or containment
Outcome, quality, integration and context verification
Regression test, policy, automation or improvement experiment
```

## Clean reconstruction

From clean checkout and empty laboratory state:

1. verify tools and laboratory scope;
2. build baseline Go/PostgreSQL product;
3. generate synthetic change/PR/defect history;
4. frame problem, outcome and hypothesis;
5. define requirements, acceptance and non-goals;
6. run research and record RFC/ADR/specification;
7. create vertical slices and migration sequence;
8. prioritize, forecast and map dependencies/WIP;
9. implement slices through atomic commits and focused PRs;
10. run multi-perspective reviews and combined-state integration;
11. run layered tests and CI gates;
12. inject and analyze one escaped defect;
13. complete docs, ownership, onboarding and handoff;
14. calculate flow/quality metrics and run improvement experiment;
15. run `engtriage` and systemic review;
16. remove temporary branches, histories, containers and databases safely.

## Complete combined exercise

Perform one scenario:

1. a vague stakeholder solution request enters the system;
2. missing baseline and conflicting quality requirements create rework;
3. decomposition produces component phases and a long-lived branch;
4. one external contract changes while integration is delayed;
5. review queue grows around one owner;
6. CI reruns a flaky test and approves stale revision;
7. a defect escapes despite high coverage;
8. urgent remediation is added without displacing planned work;
9. point forecast remains unchanged and quality/migration scope is cut;
10. `engtriage` reconstructs waiting, decisions, diffs and failed checks;
11. the change is reframed, sliced, reforecast and integrated incrementally;
12. defect class, ownership, CI and documentation are corrected;
13. metrics expose tail/rework without individual ranking;
14. a bounded improvement experiment is run and evaluated.

## Git and quality exercise

1. recover a lost commit through reflog;
2. locate a regression through deterministic bisect;
3. restructure a tangled branch into atomic commits and focused PRs;
4. resolve one review disagreement with evidence;
5. repair a mock/contract drift and one flaky CI check;
6. revert one slice while preserving compatibility;
7. prove that latest combined revision passes required checks.

## Completion criteria

The intensive is complete when:

- clean reconstruction creates the complete synthetic engineering system;
- problem, outcome, requirements and acceptance are evidence-based;
- vertical slices provide early value/learning and safe compatibility;
- research leads to an explicit decision and disposable-code boundary;
- priority, forecast and replan show uncertainty and displaced work;
- critical dependencies and queues are visible and reduced;
- Git history is coherent, recoverable and reversible;
- PRs are focused and reviews address real risk on latest revision;
- ownership has fallback and combined-state verification;
- test strategy covers critical risks at faithful levels;
- CI is deterministic, proportionate and handles flaky/infrastructure failures explicitly;
- escaped defect produces class-level recurrence prevention;
- documentation and decisions are current, findable and non-duplicative;
- onboarding/handoff is verified through independent task;
- roles and team interfaces have clear decision rights;
- metrics are reproducible, balanced and never rank individuals;
- one improvement experiment is retained, adapted or stopped through evidence;
- debt/standardization/automation decisions show net value and retirement paths;
- `engtriage` is read-only and bounded;
- all 56 faults are reproducible or have documented equivalent substitutes;
- at least 24 blind cases and two combined cases are solved;
- all 40 defense answers are evidence-based;
- cleanup finds no unintended branch, repository, container, database or generated history;
- generalized knowledge is transferred to `knowledge`.

## Delayed assessment

After 7–14 days without rereading:

1. rebuild the laboratory from clean state;
2. answer the 40 defense questions;
3. diagnose six random faults covering framing, flow, Git/review, quality, context and metrics;
4. frame and slice one new substantial change;
5. produce a probabilistic forecast and replan after disruption;
6. recover one Git history error and bisect one regression;
7. review one unfamiliar PR and classify findings;
8. design a risk-based test/CI strategy for the new change;
9. write one RFC/ADR pair and hand off an incomplete slice;
10. calculate flow/quality measures from a new synthetic history;
11. run one improvement experiment and decide retain/adapt/stop;
12. use `engtriage` without changing source data;
13. identify one ceremony/metric/tool to remove and one structural constraint to improve;
14. explain which remaining work belongs to product, architecture, delivery, operations, security or people management.

После delayed assessment уровень 3–4 можно считать закреплённым.
